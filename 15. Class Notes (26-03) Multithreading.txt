Introduction to Multithreading (Java)
========================================


--Before learning multithreading, we must understand multitasking.

--Consider the following program:

class Test {

    void fun1() { }
    void fun2() { }
    void fun3() { }

    public static void main(String[] args) {
        Test t = new Test();
        t.fun1();
        t.fun2();
        t.fun3();
    }
}


Execution Flow (Normal Execution)

First fun1() executes completely, then control comes back to the main method and then fun2() executes completely then again control comes to the main and then fun3() executes completely.

--This is called Sequential Execution.


CPU Idle Problem:
-----------------

--Suppose fun1() contains I/O operation (file reading, data transfer, network call, etc.)

--I/O operation is handled by DMA (Direct Memory Access)
--CPU does not perform data transfer(CPU assign this job to the DMA)
--During I/O operation, CPU remains idle

--This is a waste of CPU time

--This made s/w developers to think that,efficiency of the processor(CPU) would be increased if processor is made to do some other useful work during this idle state.

--The useful work is nothing but,executing other functions/methods present in the program.


--this need lead to the concept of multitasking..


--Remember that processor is also an electric circuit and at any instance of time,it can execute only one instruction.it can not execute multiple instructions simultaneously unless it is a multi-core processor.


What is Multitasking?
---------------------

--Multitasking is a technique in which CPU executes multiple tasks by switching between them very fast, giving the illusion that tasks are running simultaneously.

Important Point:

--A single CPU core executes one instruction at a time
--But switches between tasks very fast
--This is called Context Switching


--If we apply multitasking then a part of the one fun get executed then control switchs to the another fun now here also some part of second fun get executed then conrol switchs to the third fun then again control comes back to the fun1 now it continues fun1 from where it had stopped earliar.

--now when we see the output we feel that three fun executing simulteneously.

--Thus,The concept of multitasking come into existence to avoid the idle state of the CPU.



Context Switching:
------------------

--Context Switching is the process of saving the state of one task and switching the CPU to another task.

Example:

--CPU runs fun1 for some time
--Switches to fun2
--Switches to fun3
--Comes back to fun1

--So all functions appear to run simultaneously.


--In multitasking ,a part of a functionality is executed one at a time and in that part how many statement will be executed will be decided by the scheduling.



Scheduling:
----------

--It is the process in which a specific time period(Time Slice) is allocated to a functionality where the control remains in that particular functionality for that specific time period(Time Slice).

--once time period lapsed control switches another fun with another time slice and so on.

--Scheduling is supervised by the Scheduler(either OS-schedular  or Thread-scheduler).

Types of Scheduler:
-------------------

1. OS scheduler: Works on Processes (different applications)

2. Thread scheduler: Works on Threads (Single application)


---Thread-Schedular in Java is the part of the JVM, that decides which thread should run.

--In Java, thread scheduling is done by JVM Thread Scheduler, but actual execution is performed by OS.

--The time slice is very small (usually milliseconds or nanoseconds depending on OS and scheduler).. thus by the time our eye recognize the execution of one part control switches to another part of the program.




Advantage of multitasking:

--It is invented to avoid Idle states of the cpu.

1. Make the functionalities get executed independently(mostly used becoz for a small project idle state of a cpu is not a big concern).

2. Better CPU utilization
3. Faster execution
4. Improved performance
5. Reduced response time
6. Allows simultaneous tasks

Multitasking is useful for:

--Games
--Animations
--Video streaming
--Web servers
--Banking systems
--GUI applications



Multiprogramming vs Multitaksing:
---------------------------------

--Multiprogramming and multitasking are similar because both involve multiple programs sharing the CPU. However, in multiprogramming CPU switches when a program waits for I/O operations, while in multitasking CPU switches after a fixed time slice for better response time.

--In other word, Multitasking is an advanced version of multiprogramming where CPU switches between tasks after a fixed time interval instead of waiting for I/O operations.


| Term             | Trigger to Switch |
| ---------------- | ----------------- |
| Multiprogramming | I/O Wait          |
| Multitasking     | Time Slice        |





Types of Multitasking:
----------------------

--Multitasking is of two types:

1. Process based multitasking
2. Thread based multitasking



1. process based multitasking:
------------------------------

--A Process is a program under execution.

Example:

Running Chrome: Process
Running VLC player: Process
Running MS-Word : Process


--The concept of executing more than one program simultaneously which are present in the different location of RAM is known as process based multitasking.

--Here processer has to maintain the address of both the program since control has to switch from one part of ram to another.

--it incerase overhead on processor.

--Here OS (OS-scheduler) perform the scheduling.

Example

--Chrome
--VLC Player
--MS Word

--All running at same time.



2. Thread Based multitasking:
-----------------------------

--The concept of executing more than one functionality simultaneously belonging to the same memory domain is known as Thread based multitasking.

--Running multiple tasks within the same program.

--Here Java Thread-scheduler performs the scheduling.

Note: A Thread is the smallest unit of execution(lightweight sub-process) inside a process.


Example:

--Typing in MS Word while auto-save running
--Web server handling multiple users
--A program running with animation.


Process vs Thread:
------------------

| Process                         | Thread                         |
| ------------------------------- | ------------------------------ |

| Heavyweight                     | Lightweight                    |
| Separate memory                 | Shared memory                  |

What is a Thread in Java?
=========================


--A Thread is the smallest unit of execution inside a process.

--A thread is a separate flow of execution that executes some functionality of a program simultaneously with other parts of the program.



Multithreaded Application:
-------------------------



--In Java, every program/application has a default flow of execution, a defult thread, it is called as a "Main thread".

--main() method runs inside the Main thread. 

--If we can start another flow of execution(another thread) along with main thread simultenously then it is called a multithreaded application or program.
 

--When a Java program starts, one thread is automatically created called Main Thread.
--If we create additional threads, then the program becomes a Multithreaded Program.


Concurrency vs Parallelism:
===========================


Concurrency = Managing multiple tasks at the same time
Parallelism = Executing multiple tasks at the same time


Concurrency
-----------

--Concurrency is a situation where multiple tasks are in progress at the same time, but CPU executes them one by one by switching between them very fast.

--Uses context switching
--Usually happens in single-core CPU
--Tasks appear to run simultaneously, but actually they are not

Example

You are:

--Writing an assignment
--Talking on phone
--Drinking tea

--You are not doing all at the exact same time, but switching between tasks, This is Concurrency.


Parallelism:
------------

--Parallelism is a situation where multiple tasks are executed at the exact same time using multiple CPU cores.

--Requires multi-core processor
--Tasks actually run simultaneously

Example

In a restaurant kitchen:

One person making roti
One person making sabji
One person making tea

--All tasks happening at the same time, This is Parallelism.


Concurrency vs Parallelism:
---------------------------


| Concurrency               | Parallelism                 |
| ------------------------- | --------------------------- |
| Tasks appear simultaneous | Tasks actually simultaneous |
| Single CPU core           | Multiple CPU cores          |
| Context switching         | True simultaneous execution |
| Slower than parallelism   | Faster                      |
| Example: Multithreading   | Example: Multiprocessing    |




Real Life Example:
 
--One teacher teaching 3 students one by one: Concurrency

--Three teachers teaching 3 students: Parallelism



Multiprocessing
---------------

--Multiprocessing is a technique where multiple processes execute simultaneously using multiple CPU cores.

--In a multi-core CPU, each core can execute a separate process (application).
--Inside each process, multiple threads can run concurrently.
--These threads share the same memory of the process.


Note: In a multi-core CPU system, the operating system schedules processes and threads on different CPU cores. Each process can contain multiple threads, and these threads can run concurrently or in parallel depending on the number of CPU cores available.


Note: Multithreading provides concurrency and can provide parallelism on multi-core CPU.


Program vs Process vs Thread:
------------------------------

Program: A program is a passive set of instructions stored on disk.
Process: When a program is loaded into memory and starts execution, it becomes a process.
Thread: A thread is a unit of execution inside a process.




Implementing Thread in Java:
============================
1. Extending Thread Class:-
--------------------------
- Create a new class that extends the Thread class (this defines your custom thread type)
- Override the run() method inside that class (this contains the task the thread will perform)
- Create an object of your thread class (this prepares the thread instance)
- Call the start() method on that object (this starts a new thread of execution)
- The thread moves to runnable state and waits for CPU scheduling (execution timing depends on the system)
- The run() method executes when the thread gets CPU time
- After run() finishes, the thread terminates (its lifecycle ends)

Code Example:-
-------------
public class Main {
    public static void main(String[] args) {
        ThreadA ta = new ThreadA();
        ThreadB tb = new ThreadB();
        ta.start();
        tb.start();
        for(int i=100; i<121;i++){
            System.out.println(i);
        }

    }
}
public class ThreadA extends  Thread{
    @Override
    public void run() {
        for(int i=0; i<26;i++){
            char ch = (char)('A'+i);
            System.out.println(ch);
        }
    }
}
public class ThreadB extends  Thread{
    @Override
    public void run() {
        for(int i=0; i<26;i++){
            System.out.println(i);
        }
    }
}

2. Implements Runnable Interface:-
---------------------------------
- Create a class that implements the Runnable interface (this defines the task, not the thread)
- Override the run() method in that class (this contains the code to be executed by the thread)
- Create an object of your class (this prepares the runnable task)
- Create a Thread object and pass the runnable object to it (this binds the task to a thread)
- Call the start() method on the Thread object (this starts a new thread of execution)
- The thread moves to runnable state and waits for CPU scheduling (execution timing depends on the system)
- The run() method executes when the thread gets CPU time
- After run() finishes, the thread terminates (its lifecycle ends)
Code Example:-
-------------
public class Main {
    public static void main(String[] args) {
        ThreadA ta = new ThreadA();
        ThreadB tb = new ThreadB();
        Thread tr1 = new Thread(ta);
        Thread tr2 = new Thread(tb);
        tr1.start();
        tr2.start();
        for(int i=100; i<121;i++){
            System.out.println(i);
        }
    }
}

    }
}
public class ThreadB implements Runnable{
    @Override
    public void run() {
        for(int i=0; i<26;i++){
            System.out.println(i);
        }
    }

What happen when a start() method call:-
---------------------------------------
- Calling start() creates a new thread in the Java Virtual Machine.
- The thread moves from NEW state to RUNNABLE state.
- The thread becomes ready, and the thread scheduler decides when it will run.
- The JVM automatically calls the run() method internally.
- The thread executes its task independently (concurrently).
- After completing execution, the thread goes to the TERMINATED state.

Thread Priority:-
================
Thread priority is a number assigned to a thread that tells the scheduler how important that thread is compared to others. Threads with higher priority are more likely to get CPU time before lower-priority threads.

Range: 1 (MIN_PRIORITY) to 10 (MAX_PRIORITY)
Default: 5 (NORM_PRIORITY)

        (Thread.MAX_PRIORITY--10)
        (Thread.NORM_PRIORITY--5)
        (Thread.MIN_PRIORITY--1)

- What happens if you set max priority? (10)

	- The thread becomes more preferred by the scheduler.
	- It may run more often or sooner than lower-priority threads.
	- However, there is NO guarantee it will always run first.

In short:-
Setting max priority suggests the system to favor that thread, but it doesn’t strictly control execution order.

Example:-
-------

public class Main {
    public static void main(String[] args) {
        Thread1 t1 = new Thread1();
        Thread2 t2 = new Thread2();
        Thread3 t3 = new Thread3();
        
        t1.setPriority(Thread.MAX_PRIORITY);
        t2.setPriority(Thread.NORM_PRIORITY);
        t3.setPriority(Thread.MIN_PRIORITY);

        System.out.println(t1.getPriority());
        System.out.println(t2.getPriority());
        System.out.println(t3.getPriority());
        System.out.println("=====================");
        t1.start();
        t2.start();
        t3.start();
    }
}

isAlive():-
---------
isAlive() is used to check whether a thread is still running or not.

Key points:
- It returns a boolean value:
	- true → thread is still executing
	- false → thread has not started or has finished
- A thread is considered “alive”:
	- after start() is called
	- until the run() method completes

Example:-
-------
public class Main {
    public static void main(String[] args) throws InterruptedException {
        DownloadMovie dm= new DownloadMovie();
        dm.start();
        while(dm.isAlive()){
            System.out.println("Please wait movie is downloading.....");
            Thread.sleep(500);
        }
        System.out.println("Movie downloaded successfully...");
        System.out.println("You can watch the movie now.....");
    }
}

            try {
                Thread.sleep(500);

                throw new RuntimeException(e);
            }
        }
    }
}

Race condition:-
===============
A race condition happens when multiple threads access and modify the same data at the same time, and the final result depends on which thread runs first, leading to unpredictable or incorrect outcomes.

Why race condition occurs in concurrency (even without true parallelism):-
-------------------------------------------------------------------------
Even on a single core, threads don’t run all at once—they switch very fast.

Step-by-step:
- - - - - -
- Thread A starts → reads value (say count = 0)
- CPU switches to Thread B before A finishes
- Thread B reads same value (count = 0)
- Thread B increments → writes 1
- CPU switches back to Thread A
- Thread A increments its old value → also writes 1
Final result = 1 instead of 2

Example:-
--------
1.
public class Main {
    public static void main(String[] args) throws InterruptedException {
        Counter counter = new Counter();
        MyThread tr1 = new MyThread(counter);
        MyThread tr2 = new MyThread(counter);
        tr1.start();
        tr2.start();
        tr1.join();
        tr2.join();
        System.out.println(counter.returnCount());
    }
}
public class MyThread extends Thread{
    Counter counter;
    MyThread(Counter counter){
        this.counter = counter;
    }
    @Override
    public void run() {
        for(int i=0; i<10000;i++){
            counter.increment();
        }
    }
}
public class Counter {
    int count =0;
    void increment(){
        count++;
    }
    int returnCount(){
        return count;
    }
}

Lock:-
-----
A lock is a way to control access to shared resources by allowing only one thread to enter a critical section at a time, while other threads must wait until the lock is released.

How Locks Work Internally:-
--------------------------

- Every shared resource (or object) has a lock.
- When a thread enters a synchronized block/method, it tries to acquire the lock.
- If the lock is free → thread gets it immediately
- If the lock is busy → thread cannot enter
- If another thread already holds the lock, the current thread goes into a waiting/block state.
- The thread that acquired the lock executes the code that accesses shared data.
- Remaining threads keep waiting until the lock becomes available.
- After finishing execution, the thread exits the synchronized block and releases the lock.
- One of the waiting threads gets the lock and starts execution.

Think of a bathroom with one key:-
--------------------------------
- Whoever has the key can enter
- Others must wait outside
- When the key is returned, the next person can use it





1. Object Level lock
--------------------
An object-level lock is a lock that is associated with a specific object (instance), so only one thread can access that object's synchronized code at a time.

How it Works:
-------------
- Each object has its own separate lock
- When a thread enters a synchronized method/block of that object → it acquires that object's lock
- Other threads trying to use the same object must wait
- Threads can still access different objects simultaneously
- Lock is applied to a particular object, not the whole class

Example:-
--------
- Object A → locked by Thread 1
- Object B → can still be used by Thread 2
- No conflict because locks are separate per object

If an object-level lock is acquired, other threads can still access:-
---------------------------------------------------------------------
- Non-synchronized methods 
- Non-critical members (variables) directly 
- But they cannot access synchronized methods/blocks of that same object.

Step-by-step understanding:-
------------------------------
- Thread A acquires object lock
- It enters a synchronized method/block of Object X.
- Lock is now held by Thread A
- Only Thread A can execute synchronized code of that object.
- Thread B tries to access same object
	- Case 1: Calls a synchronized method
		- Blocked → must wait until lock is released
	- Case 2: Calls a non-synchronized method
		- Allowed → runs immediately
	- Case 3: Accesses variables directly
		- Allowed (but dangerous → can cause race condition)
- Object lock only controls synchronized parts, not the entire object.
- Even if one thread holds an object lock, another thread can still access a static synchronized method, because they use different locks.

2. Class Level lock
-------------------
A class-level lock is a lock associated with the class itself, not any particular object. It ensures that only one thread can execute any static synchronized method of the class at a time, no matter how many objects exist.

How It Works :
-------------
- Every class in Java has a hidden Class object. Class-level locks use this object.
- When a thread enters a static synchronized method, it tries to acquire the class lock.
- If no thread holds the class lock → thread acquires it
- If another thread holds it → thread waits
- Only one thread can execute static synchronized methods of that class at a time.
- When finished, the lock is released → next waiting thread can enter.


Key Points:
-----------
- Locks all static synchronized methods of that class, not just one method
- Does not block object-level synchronized methods (different lock)
- Prevents race conditions on shared static data

Situation:-
=-=-=-=-=-=
- Thread A acquires class-level lock by entering a static synchronized method of a class.
- There is an object of that class associated with Thread B.
- What Thread B Can Access:
	- Non-synchronized methods (static or instance)
		- No lock is needed → thread can run immediately
	- Non-static variables
		- Object-level data is independent of class-level lock
	- Static variables
		- If accessed without synchronized →  allowed
		- If accessed inside static synchronized method →  blocked until class lock is released
	- Instance synchronized methods (object-level lock)
		- Object-level lock is different from class-level lock → thread can enter

Inter-Thread Communication:-
==========================
- Inter-thread communication is a mechanism that allows threads to talk to each other and coordinate their actions while sharing resources.
- It is mainly used to avoid conflicts and ensure threads wait for conditions instead of continuously running.

Why we need Inter-Thread Communication:-
--------------------------------------
- When Threads are using shared resource
- When one Thread's needed another thread's Result.

We use following three methods for Inter-Thread Communication:-
---------------------------------------------------------------

wait():-
------
wait() is a method that makes the current thread pause and release the lock it holds, allowing other threads to run. The thread stays in a waiting state until another thread notifies it using notify() or notifyAll().

Key Points:-
------------
- Must be called inside a synchronized method or block.
- Releases the lock on the object, so other threads can enter synchronized blocks on the same object.
- The thread remains waiting until:
	- Another thread calls notify() or notifyAll() on the same object
- Used for inter-thread communication to coordinate actions.

notify():-
--------
notify() is a method used to wake up one thread that is waiting on the same object.

 How it works:-
- A thread calls wait() → it goes into waiting state
- Another thread calls notify() → one waiting thread wakes up
- The awakened thread does not run immediately — it must first re-acquire the lock

Key Points:-
- Must be used inside a synchronized block or method
- Works on the same object whose lock is held
- Wakes up only ONE waiting thread

Why we need synchronized with wait() and notify():-
===================================================
- We need a synchronized block or method when using wait() and notify() because these methods work with something called a lock (monitor) that belongs to every object in Java.
- First, understand that a thread must own the lock of an object before it can control that object. When a thread enters a synchronized block, it gets that lock.
- Now, when a thread calls wait(), it is basically saying: “I am done for now, I will release the lock and go into waiting state.”
- But a thread can only release a lock if it actually has that lock.
- So if the thread is not inside a synchronized block, it never got the lock in the first place, and therefore it cannot call wait().
- Next, when a thread calls notify() or notifyAll(), it is saying: “I will wake up one or more waiting threads.”
- But this action also needs control over the same object’s lock, because the waiting threads are associated with that lock.
- So again, the thread must be inside a synchronized block to have the authority to notify others.
- If we try to call wait() or notify() without synchronization, Java will immediately throw an error called IllegalMonitorStateException.
- This happens because Java checks whether the thread owns the lock or not, and if it doesn’t, it stops the program to prevent unsafe behavior.
-Another important reason is safety. Without synchronization, multiple threads could try to call wait() or notify() at the same time on the same object, which could create confusion, missed signals, or inconsistent data.

Deadlock in Java:-
================
Deadlock happens when two or more threads are waiting for each other to release resources.
Result: no thread can proceed → the program gets stuck.
--The program does not crash.
--No exception is thrown.
--The affected threads simply stop progressing forever.
--The rest of the application may continue, but the deadlocked part becomes unresponsive.

Simple Analogy:-
----------------
Imagine two people trying to cross a narrow bridge:
- Person A is coming from the left.
- Person B is coming from the right.
- The bridge is too narrow, so only one person can cross at a time.
- If both refuse to move, waiting for the other to go first → they are stuck. That’s deadlock!

Conditions for Deadlock
------------------------
Deadlock occurs when all four conditions exist at the same time:
- Mutual Exclusion → A resource can be used by only one thread at a time.
- Hold and Wait → A thread is holding at least one resource and waiting for another.
- No Preemption → A resource cannot be forcibly taken from a thread.
- Circular Wait → Threads form a cycle, each waiting for a resource held by the next thread.


--If even one condition is broken, deadlock cannot occur.
Note: Deadlock cannot be completely eliminated, but it can be prevented, avoided, or detected and recovered.

Solution:-
---------
- Deadlock happens when threads wait for each other in a cycle and cannot proceed.
- To handle deadlock, we can prevent it, avoid it, or detect and recover.
- Prevention: Break one of the deadlock conditions (e.g., always acquire resources in a fixed order).
- Avoidance: Check before allocating a resource that the system remains in a safe state.
- Detection and Recovery: Let deadlock occur, then detect it and resolve it by aborting threads or preempting resources.
- Prevention is simple and safe, avoidance is used for critical systems, and detection is used when deadlock is possible but cannot be prevented completely.

Wrong sequence:-
---------------
Thread1 {
	synchronized(r1){
		synchronized(r2);
	}
}
Thread2 {
	synchronized(r2){
		synchronized(r1);
	}
}
Right sequence:-
---------------
Thread1 {
	synchronized(r1){
		synchronized(r2);
	}
}
Thread2 {
	synchronized(r1){
		synchronized(r2);
	}
}
